Langflow GitHub 仓库是 [Langflow 社区](/contributing-community) 的一个组成部分。

除了对 Langflow 的一般帮助外，仓库是报告错误和请求增强功能的最佳地方，以确保 Langflow 维护者可以追踪它们。

## GitHub 问题

[Langflow 仓库中的 Issues 页面](https://github.com/langflow-ai/langflow/issues)积极更新错误和功能请求。

:::tip
推广请求或错误的最佳方式是对现有问题进行评论。
高度活跃的问题更可能获得贡献者的关注。

在报告错误或提交功能请求之前，请搜索现有的类似问题。
使用 [Langflow 仓库的标签](https://github.com/langflow-ai/langflow/labels)来帮助过滤您的搜索。
:::

## GitHub 讨论

如果您需要对您的代码或 Langflow 的一般帮助，您可以访问 [Langflow GitHub 讨论页面](https://github.com/langflow-ai/langflow/discussions)或通过其他 [Langflow 社区](/contributing-community)渠道联系。

Langflow 团队不通过电子邮件提供个人支持，团队认为公开讨论由于其可发现性而帮助更多用户。

## 社区准则和提示

由于 Issues 和 Discussion 页面是公开的，Langflow 团队请求您在提交问题和 issue 时遵循这些准则：

* **提供尽可能多的细节**：仅仅说明一个功能不起作用并不有帮助。Langflow 团队需要细节来重现和发现问题。
* **解释确切出了什么问题**：包括错误信息和您的代码如何失败的描述，而不仅仅是它失败的事实。
* **回溯您的步骤**：解释错误之前发生了什么，您期望发生什么而不是错误，以及您最近进行的任何更改，如升级 Langflow 或依赖项。
* **描述您的环境**：包括您的操作系统、Langflow 版本、Python 版本以及任何其他可能对问题有影响的环境相关细节。
* **包含失败代码的片段**：务必省略任何敏感值，并只提供与失败相关的部分，而不是整个脚本。提供代码片段使得重现错误、故障排除和提供具体建议变得更加容易。
  * 如果您的提交包括长段代码、日志或追踪，请将它们包装在 [details 标签](https://developer.mozilla.org/en/docs/Web/HTML/Element/details)（`<details> PASTE CODE HERE </details>`）中以折叠内容并使您的提交更易阅读。
* **省略敏感信息**：除了您公开 GitHub 配置文件中可用的信息外，不要包含敏感或个人身份数据，如安全密钥、全名、个人身份号码、地址和电话号码。
* **保持友善**：尽管任何软件的错误都可能令人沮丧，但请记住您的信息是由想要帮助的真实人员阅读的。虽然您不必过分甜腹，但也没有必要粗鲁地获得支持。
  * 您的 issues 和讨论附加到您的 GitHub 帐户，互联网上的任何人都可以阅读，包括当前和潜在的雇主和同事。
  * Langflow 仓库是一个公开的 GitHub 仓库，因此遵循 [GitHub 行为准则](https://docs.github.com/en/site-policy/github-terms/github-community-code-of-conduct)。
